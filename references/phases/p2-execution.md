# Phase Module · P2 Execution

> The execution phase delivers increments/outcomes per the **baselined plan**. It runs **concurrently** with **P3 Monitoring** within the `operational` state
> (execution is "doing," monitoring is "watching"); it is not sequential (execute first, then monitor).
> State-machine placement: `执行` ⊆ `operational` (must first pass the G1→2 control gate, and waterfall/hybrid must have run `baseline.py --freeze`).
> The hard gate to enter this phase: G1→2 (see `p0-p1-initiation-planning.md §6` and `scripts/gate_engine.py`).

## 1. Objectives

Deliver acceptable increments or outcomes per plan/commitments; maintain baseline comparison; control scope/schedule/cost variance.

## 2. Key Activities (Methodology Adaptation)

| Methodology | Execution Activities | Cadence |
|------|----------|------|
| waterfall | Advance by WBS package; quality-plan checkpoint entry/exit; changes via CCB | Advance by dependency within stage; weekly/biweekly status |
| agile (Scrum) | Sprint planning → daily standup → Sprint review (demoable increment) → retrospective; strict DoD | Fixed Sprint (1–4 weeks) |
| agile (Kanban) | Kanban flow; limit WIP; measure Lead Time / flow efficiency | Continuous flow |
| iteration | Iteration planning → iteration execution (time-boxed) → iteration review → iteration retrospective; scope locked within iteration | Fixed iteration (e.g. 2 weeks) |
| hybrid | Micro-layer executes per its methodology; macro-layer checks at alignment review points whether micro-layer increments meet gate entry | Macro low-freq / micro high-freq + alignment review |
| program | Each component executes independently; portfolio layer continuously updates dependency map/portfolio board to drive coordination | Portfolio cadence |

## 3. Required Deliverables (Templates)

- **Increment/outcome**: Iteration/Sprint output, `agile/sprint_plan`, `iteration/iteration_plan`, `agile/retro`, `agile/burndown`, `iteration/iteration_review`
- **Change control**: `common/change_request` + `common/change_log` (any scope/schedule/resource change must go through the formal channel, writing back to `project.yaml`)
- **Operations startup**: `common/control_register` (generated by baseline.py, defines routine control checklist/frequency/owners)
- **Demo/acceptance artifacts**: Demoable increments per methodology (review/demo conclusions)

## 4. Entry Criteria (Entry · G1→2 Control Gate)

- `lifecycle_state` ∈ {planning, review, baselined} and entered via approval.
- Hard gate (automated checks, all required):
  - `consistency_check.py --project` **exit 0** (no fatal issues in plan baseline).
  - waterfall/hybrid: **`baseline.py --freeze` already run** (`baseline.file` exists).
- Approver: sponsor (see `p0-p1-initiation-planning.md §6` `gate_engine.py --to 执行 --approve`).

## 5. Exit Criteria (Exit · Ready to Enter Monitoring/Closeout)

- Advance per baseline/commitments, variance within thresholds (schedule slip < `schedule_slip_pct`, SPI/CPI ≥ 0.95).
- `control_register` established, operations control loop started (`control_engine.py` runs per `control.cadence`).
- Major variance/changes have gone through CCB and been written back.
- Note: Monitoring runs concurrently with execution; no need to "finish execution" before entering monitoring; both reside in `operational`.

## 6. Phase-Gate Approval (Gate · G2→3 Soft Gate)

- **Gate**: G2→3 Execution→Monitoring (soft gate, marks monitoring cadence within operational, does not change state machine).
- **Approver**: PM.
- **Commands**:

```bash
SKILL_DIR=<this skill directory>
python3 $SKILL_DIR/scripts/gate_engine.py --project /workspace/<slug>/project.yaml --to 监控 --approve "李四(PM)"
```

## 7. Recommended Scripts

- `control_engine.py`: Periodic inspection during operations (against baseline), RED escalation exits 1.
- `evm.py`: Establish and roll EVM baseline (pv/ev/ac) during execution, compute CPI/SPI.
- `schedule_health.py`: waterfall/hybrid tracks critical path/float.
- `render.py` / `render_docx.py`: Produce increments, demo artifacts, formal documents.
- `gate_engine.py --to 监控`: Soft-gate markers for monitoring cadence.

## 7.1 Operational Artifact Guardrail (OAG) — 每次动作必须刷新交付物

> **Iron Rule #12，不可妥协。** 本阶段（operational）曾发生 `raid_log` / `risk_register` / `portfolio_dashboard`
> 落后于事实源的护栏违规。每次执行动作都必须闭环交付物。

**强制后置步骤（每次 operational 动作后）**：
1. 修改 `project.yaml`（状态 / EVM / RAID / WBS / actuals）后，**重渲染**受影响交付物：
   - RAID 变更 → `python3 $SKILL_DIR/scripts/build_subproject.py --project <yaml> --only raid`
   - 状态/EVM 变更 → 重渲染 `status_report` / `portfolio_dashboard`（程序层）
2. 手工/外部渲染的文档产出后：`python3 $SKILL_DIR/scripts/artifact_guard.py --project <yaml> --stamp <key>`
3. 校验：`python3 $SKILL_DIR/scripts/artifact_guard.py --project <yaml>` → **必须 exit 0**；非零则回到第 1 步。
4. `control_engine.py` 周期巡检会内置 OAG 检查，漂移即 RED 升级（exit 1）。

**违规后果**：交付物漂移 → `control_engine` RED / 收尾门（G3→4）硬准则不通过 → 禁止收尾。

## 8. Handoff

- This phase is the "execution track" of the **operational dual-track**: concurrent with the **P3 monitoring track** (`monitoring-agent` periodically runs `control_engine.py`) in `operational`,
  sharing `project.yaml`+`baselines/`, with zero field conflicts (execution track writes `actuals`/`wbs_progress`/deliverables, monitoring track writes controls/status reports).
  Variance/risk is continuously pulled back to baseline by the monitoring track, and corrective actions flow back to this track. See `references/orchestration.md §3.4` for dual-track orchestration.
- After meeting closeout exit (see `p4-closeout.md §5`), flip `closed` via the **G3→4 closeout gate**.
